# DGBook P1 完整数字教材样张需求基线

**状态：** 已确认，替代 2026-07-14 规格中与本文件冲突的内容  
**确认日期：** 2026-07-15  
**唯一产品目标：** 用一个可重复演示的“一师三生 + 能力图谱”闭环，证明 P1 是完整数字教材样张，而不是若干页面或数据看板。

## 1. 演示必须证明的闭环

```text
教师登录 -> 授课工作台 -> 继续/开始 P1 N02 课堂
                                   |
学生登录 -> 学习首页 -> 主动进入课堂跟随 / 返回个人自学
                                   |
P01 室内信息采集 -> P02 室外信息采集 -> P03 投诉信息采集
                                   |
微练习 -> 正式测试 -> 结构化专业产出 -> 轻量教师认证
                                   |
三个任务产出 -> 《5G网络信息采集成果包》 -> 能力图谱同步
```

样张默认只有 1 名教师、3 名学生，便于人工验证；数据库、查询和 UI 不得把 3 或 24 写成容量上限。

## 2. 角色默认入口

### 学生 `/student/home`

登录后始终先进入学习首页，不自动跳入课堂。首屏在 30 秒内回答：

1. 我正在学什么。
2. 为什么学。
3. 下一步做什么。
4. 做到什么算完成。

首屏必须显示当前项目、任务、能力节点、完成标准、唯一主按钮，以及“查看其他任务”和“课程能力图谱”次级入口。课堂进行中时，主按钮变为“进入正在进行的课堂”；学生主动加入后才跟随教师，退出后返回原个人游标。

### 教师 `/teacher/workbench`

登录后进入授课工作台，显示当前课程、演示班级、最近授课位置、继续授课、开始新课、待批阅专业产出、班级薄弱点和图谱入口。教师不理解图谱也能在两次点击内进入任一 P1 N02 授课页；P1T1-N02 必须满足该验收。

## 3. P1 完整项目

P1 按严格前后关系呈现：

| 任务 | 名称 | 深度核心节点 | 任务专业产出 |
|---|---|---|---|
| P01 | 室内环境信息采集 | P1T1-N02 设备拓扑 | 室内设备与链路证据表 |
| P02 | 室外环境信息采集 | P1T2-N02 天线姿态 | 室外站点与覆盖采集表 |
| P03 | 投诉信息采集 | P1T3-N02 复现场景 | 投诉信息调查单 |

三项任务必须全部可进入、可学习、可练习、可测试、可提交，不得使用占位页或“未开放”代替完成。三个任务产出自动汇总为《5G网络信息采集成果包》，项目页显示任务状态、产出版本和项目完成状态。

P2 及以后只在课程能力图谱保留完整轮廓，明确显示“后续开放”；不得生成虚构成绩、状态或学习页面。

## 4. 教材深度

三个 N02 都是深度核心节点，统一保留“问题、看图、步骤、纠偏、练习、产出”六段连续结构。每个 N02 必须包含：

- 案例背景、任务问题、必要前置知识和术语解释。
- 带标注专业图，说明证据在哪里、对象是谁、连接或因果方向是什么。
- 明确判断依据和可复述的逐步推理过程。
- 至少 2 个完整正例和 2 个完整反例。
- 基础、应用、迁移三级练习。
- 按错误原因给出的反馈、改正路径和可重试机制。
- 迁移任务、结构化记录模板和评价量规。

其他节点也必须能够独立学完，最低包含连续案例、必要术语、关系图、判断步骤、1 个正例、1 个反例、可重试微练习、针对性反馈和结构化节点记录；但不强求与 N02 相同的媒体密度。

关闭教师讲解、语音和虚拟人后，正文仍须独立成立。教材保持单向讲授，不新增聊天导师、学生对话、讨论面板或圆桌流程。

## 5. 自学与课堂跟随

- 自学模式展示完整教材、自由前后阅读、术语查询、案例、练习、反馈、重试和专业产出。
- 课堂跟随只展示教师当前单元、课堂位置、教师任务、课堂活动和“返回完整自学”入口。
- `selfStudyCursor` 按学生与节点保存；`classroomCursor` 按课堂会话保存。
- 教师切页更新课堂游标；只有已主动加入课堂的学生同步变化。
- 课堂命令不得覆盖学生个人阅读游标。
- 单实例样张优先使用 SQLite 修订号加 1 秒课堂轮询；后台页面降为 15 秒。SSE 仅作为可选优化，不是完成前置条件。

## 6. 三名学生演示状态

自主学习种子错开：

- 学生一位于 P01。
- 学生二已完成 P01，位于 P02。
- 学生三已完成 P01、P02，位于 P03。

教师开始同一课堂后，三人可以共同跟随当前课堂页面；退出课堂后分别恢复原自主进度。种子必须来自 SQLite 事件和记录，不得由组件硬编码。

## 7. 统一状态机与门禁

唯一状态集合：

```text
未解锁 -> 可学习 -> 学习中 -> 微练习通过 -> 正式测试达标
-> 专业产出已提交 -> 待教师复核 -> 退回修订 / 教师认证 -> 能力达成
```

节点配置显式声明是否需要微练习、正式测试、专业产出和教师认证。N02 承担正式节点测试；N04 承担结构化任务产出与一次轻量教师认证。不建设补考、多级审批、教务回写和复杂改分流程。

页面和所有写 API 使用同一服务端门禁：

- `open`：允许加载教材与提交。
- `locked`：只显示缺失前置条件。
- `not-open`：明确显示后续开放。
- `not-found`：明确显示节点不存在。

后三类不得加载教材正文、练习、测试或提交组件；任何未知节点不得回退 P1T1-N01。

## 8. 专业产出与评分

三个任务的专业产出全部使用站内结构化表单，支持保存草稿、提交、查看版本和轻量教师认证；样张不依赖 Word、Excel 或附件上传。

统一口径：

- 节点测试最高分：该 N02 正式测试历次最高分。
- 任务综合分：N02 正式测试 40% + N04 专业产出量规 60%。
- 项目综合分：P01、P02、P03 三个任务综合分等权平均。
- 必要产出未形成时显示“尚未形成”，不以 0 或模拟分替代。
- 正式测试达到 80 分只代表“测试达标”，不代表教师认证或能力达成。

学生、教师、投屏和图谱不得使用无上下文的“成绩”标签。

## 9. 单一数据源

SQLite 是账号、会话、学生进度、测试尝试、产出、认证、课堂位置和四端统计的唯一事实源。课程定义和教材正文可以是版本化静态文件，但所有可变化事实必须由 SQLite 事务写入和统一读模型投影。

学生、教师、投屏和能力图谱的同一快照必须具有相同的 `snapshotVersion`、课堂修订号、班级人数、提交数、达标数、状态和分数。投屏裁剪个人隐私字段，但不得另算统计。

## 10. UI 与可用性

直接对齐 `docs/design/image2/` 的 V4 参考：深海军蓝工程空间、青色焦点、绿色达成、黄色风险、红色异常；真实图标，半径不超过 8px，明确边界，不使用紫色渐变和无意义霓虹。

1440×900 与 1920×1080 必须无横向溢出。图谱边必须连接真实源/目标节点，标签不得压线。学生和教师首屏保持一个主行动；长教材正文优先保证可读性。

## 11. 工程清理

`apps/web` 是唯一正式前台。清理分三类执行：

1. 直接白名单删除可重建项：`.next`、缓存、`__pycache__`、临时日志、旧 Playwright 会话、旧截图和旧发布包。
2. 验证引用并迁移后移出活动工程：旧 Astro `site` 应用、`studio`、`OpenMAIC`、旧 `deploy:sample:*` 链和过期文档；`site/public/media` 在迁移完成前必须保留。
3. 必须保留：`content/5g/5g.docx`、导入器、`textbook` 权威生成物、当前 Web 依赖包、数据库迁移和必要媒体。

当前 Git 无可信基线，禁止 `git clean -fd`、`git reset --hard` 和基于“未跟踪”状态的批量删除。每次清理使用显式路径白名单，并在删除前跑引用扫描、删除后跑完整门禁。

## 12. 阶段完成合同

每个纵向切片必须依次完成：聚焦测试、`pnpm web:test:unit`、`pnpm web:typecheck`、`pnpm web:check-structure`、Node 20 生产构建、浏览器截图、source release 审计、远端原子部署、`/api/build-info` 校验和公开 URL 冒烟。

Windows standalone 最终软链接复制的 EPERM 不得冒充产品失败；权威生产构建在远端 Linux 完成。远端失败不得切换 `current`，并保留上一 release 在线。

- 1 名教师和 3 名学生都能真实登录。
- 学生、教师和图谱默认入口符合本规格。
- P01、P02、P03 全部可学完，三个 N02 达到深度标准。
- 三份结构化产出可提交、认证并汇总成果包。
- 两种学习模式及双游标通过跨浏览器验证。
- 状态、路由、API 和四端快照一致。
- Image2 双视口验证通过。
- 活动工程无已确认的重复前台、临时产物和失效发布入口。
