# DGBook P0 真实性闭环设计

## 目标

在不重构学生首页、教师工作台和课程能力图谱视觉的前提下，把 P01 的学习、岗位练习、正式测试、专业产出、教师确认、成果包和三端课堂协同变成可由真实事件证明的完整样张；同时补出面向匿名访客的只读平台总览与资源生产链展示。

本设计覆盖 P0-1 至 P0-9。实施拆成两个可独立验收的阶段：第一阶段交付 P0-1 与 P0-2 至 P0-6；第二阶段在真实测试与成果契约稳定后交付 P0-7 至 P0-9。

## 已确认边界

- `/` 继续作为学生和教师的唯一登录入口，不改变两个角色的默认首页。
- `/platform`、`/resources`、`/governance`、`/delivery` 仅提供匿名、脱敏、GET-only 的 P1 生产链展示，不提供上传、在线生成、审核、发布或完整资源包下载。
- 匿名区只公开平台说明、资源卡、缩略图、短样片和脱敏状态。完整教材、教师讲稿、答案、测试、成果内容、学生证据和成绩必须登录后才可访问。
- 当前仍只有 `student` 与 `teacher` 两种账号；不新增平台运营者角色。
- 证据挂接使用内置现场证据库，不建设本地文件上传。
- 三名学生固定为：学生一全新学习；学生二处于退回修订；学生三具有完整达成案例。学生二、三所有预置事实必须显示“演示数据”，且可被重置。
- 正式测试未达标后生成定向诊断；学生完成指定再学活动后自动解锁重试。保留全部历史并取最高有效分，不建设三次锁定、补考申请或教师审批。
- P01 正式测试达标线保持 80 分；任务综合分保持 N02 正式测试 40% + N04 专业产出量规 60%。测试达标不等于教师确认或能力达成。
- N01、N03 的岗位活动真实通过后可显示该节点达成；P01 任务能力达成必须等待 N04 成果被教师确认。
- P01 两课时采用 `2 × 45 分钟`：第一课时 N01 + N02 + 正式测试，第二课时 N03 + N04 + 成果提交与项目收束。
- 浏览器统一课堂会话是 Demo 的同步主链路；外部课堂助手降为可选扩展。投屏上一页/下一页改变全局课堂游标，返回只退出投屏。

## 总体架构

```text
权威教材内容 + 内置证据库
          |
          v
岗位活动定义 -> 作答尝试/活动产物 -> N04 自动预填成果表
          |                    |               |
          v                    v               v
定向反馈与重试          服务端正式测试判分    版本/证据/批注
          \____________________|_______________/
                               v
                    SQLite 权威事件与版本记录
                               v
              学生 / 教师 / 投屏 / 图谱统一投影
```

禁止页面从名称、预置分数或本地浏览器状态推断完成。所有完成、分数、提交、退回与确认均来自 SQLite 中可追溯的真实事件。

## P0-1：公开平台与资源生产链

### 页面

- `/platform`：展示“输入材料 → 内容诊断与检索补强 → 课程能力图谱 → 资源生成 → 审核治理 → 数字教材 → 教学应用 → 数据回收”。
- `/resources`：展示固定 P1 样例的材料、知识原子、资源类型、教材挂接位置和缩略预览。
- `/governance`：展示脱敏后的质量门禁类别、通过/需修订状态和版本流，不公开答案、原始路径或内部日志。
- `/delivery`：展示出版社资源包的清单摘要与数字教材直接挂接摘要，不提供完整包下载。

公开 API 只返回白名单 DTO：`id`、`title`、`kind`、`status`、`summary`、`thumbnailUrl`、`outputMode`。不得序列化完整教材、`expectedEvidence`、`correctModel`、量规答案、教师讲稿或学生数据。

## P0-2：岗位化学习活动

内容模型新增判别式 `ActivityDefinition`，至少包含：

```ts
type ActivityKind =
  | 'scope-classification'
  | 'evidence-classification'
  | 'link-reconstruction'
  | 'structured-record'
  | 'four-state-judgement'
  | 'defective-sheet-revision';

interface ActivityDefinition {
  id: string;
  nodeId: string;
  kind: ActivityKind;
  materials: ActivityMaterial[];
  prompt: string;
  responseSchema: ActivityResponseSchema;
  feedbackRules: ActivityFeedbackRule[];
  remediationTarget: { nodeId: string; sectionId: string };
}
```

- N01：对模拟对象进行采集范围分类，必须处理纳入、排除和排除理由。
- N02：连续完成证据分类、链路重建和结构化记录；位置、身份、方向证据不可互换。
- N03：逐条判断“满足、异常、待复核、无权操作”，并说明判断依据和后续动作。
- N04：检查并修订一份照片编号、字段来源和链路关系存在缺陷的成果表。

每次作答保存 `studentId`、`activityId`、`response`、`result`、`artifact`、`origin`、`attemptedAt`。针对性反馈必须来自活动规则，重新作答产生新尝试，不覆盖历史。

## P0-3：真实室内设备与链路证据表

N04 使用任务级权威成果定义，不再从 N02 文本临时拼字段。成果至少包含：站点与机房、采集范围、位置证据、设备身份、双端端口、连接方向、照片索引、证据缺口、风险与复核结论。

N01 至 N03 通过后的结构化活动产物自动预填，并记录 `sourceNodeId + sourceAttemptId`。学生可以修订预填值，但不能抹掉来源。字段证据通过 `evidence_links(outputId, version, fieldKey, evidenceId)` 挂接到内置证据库。

复用现有不可变版本与 `draft/submitted/returned/verified` 存储状态。界面六态从状态、提交次数和复核历史确定性投影：

```text
编辑中 -> 已提交 -> 教师退回 -> 修订中 -> 再次提交 -> 教师确认
```

空表不能提交；没有提交事件不能显示已提交；没有教师确认事件不能显示教师确认或任务能力达成。

## P0-4：服务端判分正式测试

学生从独立入口进入 `/learn/P1T1-N02/test`。正式测试包含四个分项：证据分类、链路重建、错误成果修订、职业结论表达。

服务端下发不含答案的题面、题目版本和 attempt token。客户端提交具体答案，禁止提交最终分数。服务端持有答案与量规，完成判分后一次性写入：总分、四项分数、答案快照、错误类型、诊断、再学目标、来源和完成时间。

客户端 bundle 不得包含 `correct`、`targetId`、答案解析或教师量规。未达标结果页必须链接到具体节点与教材段；完成指定再学活动后才能创建下一次正式尝试。

## P0-5：真实状态与演示来源

保留现有统一学习状态机，但推进条件改为具体事件与 `requiredActivityIds`，不能用任意课堂提交或页面打开代替岗位活动通过。

所有预置尝试、成果、复核和冻结分增加 `origin: 'demo' | 'user'`。演示种子必须通过与真实用户相同的 schema 和领域校验生成完整内容，禁止原始插入空壳 `verified` 成果。用户重新完成流程后，个人当前状态优先展示 `origin=user` 的事实；历史演示记录仍保留来源标签。

学生一不含正式测试分、成果或确认。学生二包含完整 V1、字段证据、教师退回意见和待修订状态。学生三包含完整作答、正式测试诊断、成果版本、量规与教师确认；所有预置内容标记“演示数据”。

课堂正式测试统计必须绑定具体 assessment instance 与开启时间，不得把历史或演示种子成绩混入当前提交数。

## P0-6：可打开的成果包

新增任务成果详情页，展示：当前成果字段、字段证据、教师字段级批注、V1/V2逐字段差异、正式测试分项诊断、对应能力节点与来源标签。成果包卡片链接到真实详情页。

没有实际成果内容时显示“尚未形成”；只有完整演示内容时显示“演示数据”，不能称为可交付成果。项目级成果包只在 P01、P02、P03 均形成可追溯成果后显示完成。

## P0-7：P01 两课时授课包

新增独立 `TeacherLessonPackage`，每页具有 `duration`、`projectorContent`、`teacherNarration`、`question`、`modelAnswer`、`commonErrors`、`followUps`、`studentAction`、`transition`。教师不再从学生单元动态拼三句通用讲稿。

教师授课包复用岗位活动材料、正式测试诊断类型和成果表量规，确保投屏、讲解、提问和学生动作语义一致。

## P0-8：三端课堂协同

- 浏览器课堂连接状态决定共享操作是否可用；离线时显示重新连接并阻止共享状态写入。
- 外部助手状态独立显示为可选设备扩展，不再把“助手离线”表达为课堂同步不可用。
- 课堂活动和正式测试提交数为 0 时，UI 与服务端都禁止进入讲评。
- 投屏提供全屏、上一页、下一页和返回；翻页使用相同的 revision/CAS 会话命令。
- 自学学生只收到课堂有新位置的提示，不覆盖个人游标。
- 讲评只投影匿名错误类别与人数，不包含学生姓名、ID或答案原文。

## P0-9：图谱只修挂接

保留现有 D3 图谱视觉、层级、缩放和边。图谱活动节点新增明确目标：`learn`、`formal-assessment`、`professional-output`、`task-score`。

学生正式测试进入独立测试；专业产出进入真实成果；教师对应进入启动测试或批阅队列。学生图谱显示个人任务综合分，教师图谱显示班级任务综合平均分。所有分数与状态均来自权威快照，并保留 `demo/user` 来源。

## 错误处理与安全

- 匿名写操作统一返回 401；角色越权返回 403；锁定节点返回带前置条件的专用页面。
- attempt token 只可使用一次并绑定学生、节点、题目版本和 assessment instance。
- 答案不进入公开 API、`public/` 目录或客户端 bundle。
- 版本冲突继续使用现有 `expectedStateRevision` 返回 409，不静默覆盖。
- 演示重置只作用于三名演示学生，不清空课程定义、媒体或生产数据库。

## 验收策略

第一阶段必须有自动化证据证明：匿名预览白名单；四种岗位动作与针对性反馈；服务端判分且伪造 `score=100` 无效；N04 自动预填、证据挂接、退回修订与确认；三种学生演示状态；当前课堂统计不混入历史分；成果详情能打开字段、证据、差异和诊断。

第二阶段必须证明：两课时所有授课字段完整；教师翻页同步投屏和跟随学生且不覆盖自学；0提交讲评被双重门禁；匿名错因不泄露身份；图谱四类目标路由和真实状态一致。

每阶段运行受影响单测、`pnpm web:typecheck`、`pnpm web:check-structure`、生产构建和真实浏览器旅程；最终运行 `pnpm qa:gates`，再分阶段部署与现网冒烟。
