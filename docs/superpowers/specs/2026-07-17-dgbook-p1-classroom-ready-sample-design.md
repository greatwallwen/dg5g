# DGBook P1 四课时真实教学样张设计

**状态：** 已批准  
**批准日期：** 2026-07-17  
**适用范围：** 当前 DGBook 5G 数字教材样张  
**替代关系：** 本文补充并覆盖既有规格中与课堂生命周期、双游标、活动等价、跨任务解锁、正式测试计时和 P01 课时安排冲突的内容；未冲突部分继续有效。

## 1. 产品目标

把当前 P1 从“可以浏览的演示页面”收敛为一套能够由一名教师和三名学生真实完成的四课时数字教材样张。样张必须证明以下闭环，而不是只展示页面或预置数字：

```text
教师准备并开始课时
        ↓
三名学生自主选择课堂跟随或完整自学
        ↓
真实岗位活动 → 针对性反馈 → 重试 → 微练习通过
        ↓
服务端正式测试 → 分项诊断 → 定向再学
        ↓
N04 专业成果 → 提交 → 退回 → 修订 → 教师确认
        ↓
下一任务开放 + 任务正式分形成 + 图谱统一回流
        ↓
P01、P02、P03 汇总为《5G网络信息采集成果包》
```

最终验收对象固定为一名教师和三名学生。`3` 是当前样张的默认演示规模，不是系统容量上限；页面、查询和领域模型不得把它写成不可扩展的业务常量。

## 2. 方案选择

### 2.1 采用方案：SQLite 真实事实源与统一纯投影

保留现有 SQLite、领域服务、快照投影和 Next.js 页面结构。补齐少量表与字段，把目前散落在前端、演示种子和课堂助手中的状态判断集中到服务端。学生端、教师端、投屏端、项目页和能力图谱只读取同一事实源生成的角色裁剪快照。

### 2.2 不采用的方案

- 不采用仅修改按钮和文案的 UI 修补。它不能消除虚假完成、提交数冲突和 URL 绕过。
- 不采用浏览器本地状态作为权威。它无法在刷新、换端和三人课堂中保持一致。
- 不采用 WebSocket、Redis、完整事件溯源或通用工作流引擎。四课时样张使用 SQLite 加活动态短轮询即可，新增基础设施会延缓真实闭环。

## 3. 范围与课时结构

P1 固定由三个任务、十二个节点和四个 45 分钟课时组成。

| 课时 | 教学作用 | 内容与闭环 | 支架强度 |
|---|---|---|---|
| P01 第1课时 | 示范建模 | N01 采集范围分类；N02 位置、身份、方向证据框架；基础活动 | 完整标注、显性推理、完整示例和即时提示 |
| P01 第2课时 | 独立应用 | N02 应用与迁移活动、正式测试；N03 四态判断；N04 缺陷成果修订与室内成果提交 | 保留检查表，撤掉完整答案与固定推理顺序 |
| P02 第1课时 | 近迁移 | 室外站点与覆盖材料中的 N01—N04、正式测试和室外成果提交 | 只保留证据类别提示与缺口检查表 |
| P03 第1课时 | 综合迁移 | 投诉场景中的 N01—N04、正式测试、投诉调查单和项目成果包收束 | 只保留成果格式、合规边界和评价标准 |

每个课时是独立且不可变的 `lessonRun`。已结束课时只读；下一课创建新的运行记录，不复用或重置上一课的活动、测试和学生结果。

教师授课包页面规模固定为 P01 `6 + 6` 页、P02 `6` 页、P03 `6` 页。每页必须提供建议时间、投屏内容、教师讲解、提问、结合案例的典型答案、常见错误、追问方式、学生动作和过渡语。教师不另做 PPT 也能完成授课。

## 4. 学生学习路径

### 4.1 首页双上下文

学生首页不能再把课堂位置当作个人进度。它同时展示：

- `教师当前讲到`：课时、节点、页码、当前活动和课堂状态。
- `我个人学到`：个人节点、六段位置、已通过活动数和下一项要求。
- `完成标准`：由节点政策和真实完成事实生成。

主行动按情境保持唯一：

- 活动课堂存在且学生可加入：`继续课堂学习`。
- 没有活动课堂：`继续个人学习`。
- 没有活动课堂且成果被退回：`修订专业成果`。

次级入口保留个人学习、其他任务和课程能力图谱。课堂暂停时不得显示可用的课堂主按钮；首页恢复以个人进度为主路径。

### 4.2 双游标与模式切换

系统永久区分两类位置：

```ts
type SelfStudyCursor = {
  studentId: string;
  nodeId: string;
  sectionId: 'problem' | 'figure' | 'steps' | 'correction' | 'practice' | 'output';
  updatedAt: string;
};

type TeachingCursor = {
  lessonRunId: string;
  lessonId: string;
  taskId: string;
  nodeId: string;
  pageId: string;
  pageIndex: number;
  phase: 'lecture' | 'question' | 'practice' | 'assessment' | 'review' | 'close';
  revision: number;
  updatedAt: string;
};
```

- 教师翻页只修改 `TeachingCursor`。
- 跟随模式学生在两秒内同步到教师的精确 `pageId`。
- 自主模式学生只收到“课堂位置已更新”提示，不自动跳页。
- 学生主动选择“回到教师当前页”后才恢复跟随。
- 返回完整自学时恢复原 `SelfStudyCursor`。
- 不要求恢复像素级滚动位置，恢复到六段中的准确段落即可。

### 4.3 阅读轨迹与能力证据分离

前四段由学生明确完成当前段后记录阅读轨迹；它们不能直接产生能力达成。`practice` 只由必要活动真实通过产生，N04 的 `output` 只由有效成果提交产生。禁止用一次“完成节点”批量生成所有段落完成事件。

SQLite 必须恢复上次段落、已通过活动 ID、尝试次数、最近错误类型和纠正路径。退出登录、刷新和课堂/自学切换后不能从 `2/3` 退回 `0/3`。

## 5. 统一岗位活动

### 5.1 同一活动、两种交付渠道

课堂和自学使用相同 `canonicalActivityId`、材料、响应 schema、服务端评价器和通过阈值。它们只在交付渠道上不同：

```ts
type ActivityAttempt = {
  attemptId: string;
  studentId: string;
  nodeId: string;
  canonicalActivityId: string;
  deliveryChannel: 'self-study' | 'classroom';
  lessonRunId?: string;
  pageId?: string;
  answers: unknown;
  passed: boolean;
  mistakeCodes: string[];
  fieldFeedback: Record<string, string>;
  correctionPath?: { nodeId: string; sectionId: string };
  attemptNumber: number;
  origin: 'user' | 'demo';
  attemptedAt: string;
};
```

任一渠道的真实通过记录都满足对应必要活动，另一端不得要求重复完成。课堂中的删减演示如果不使用同一评价器，必须配置 `completionCredit: 'none'`。

### 5.2 四类岗位动作

- N01：对模拟对象进行采集范围分类，明确纳入、排除和理由。
- N02：依次完成证据分类、链路重建和结构化记录；位置、身份、连接方向证据不可互相替代。
- N03：逐条判断满足、异常、待复核、无权操作，并给出依据与下一步动作。
- N04：检查和修订存在字段来源、照片索引、链路关系或合规边界缺陷的成果表。

反馈至少返回 `passed`、`mistakeCodes`、`fieldFeedback`、`correctionPath` 和 `attemptNumber`。纠正路径必须深链到准确节点和段落；重试保留历史尝试与反馈。

## 6. 课堂授课与三端协同

### 6.1 正交课堂状态

课堂生命周期：

```text
preparing → active ⇄ paused → closed
```

课堂生命周期、授课页游标、普通活动窗口、正式测试窗口、媒体播放位置和同步健康必须分别存储，不能互相冒充。

- `preparing`：教师预览课时并检查服务器、投屏和学生设备。
- `active`：允许翻页、发起活动和测试。
- `paused`：冻结授课页推进；普通活动保留草稿。
- `closed`：终态，只读；课后批阅继续可用。

课堂暂停不自动暂停已经开始的正式测试。遇到全班技术故障时，教师必须使用独立的“暂停测试”操作；该操作对全部考生生效并写入审计记录。

### 6.2 真实设备在线

已登录的真实学生、教师和投屏浏览器发送心跳。建议状态口径：五秒内在线、五至十五秒连接不稳定、超过十五秒离线。活动课堂使用约一秒轮询读取权威修订；非活动页面降低频率。

现有外部课堂助手退出核心权威链路，只保留为可选的“演示设备模拟器”。助手离线不能阻止教师本地查看教材、翻页和投屏，也不能伪造学生在线或提交。

降级规则：

- 单个学生离线不影响教师、投屏和其他学生。
- 学生重连后，跟随模式追到最新教学页，自主模式保持个人位置。
- 投屏离线时停留最后成功页面并显示离线横幅；重连后读取最新权威页。
- 教师与服务器断线时仍可阅读教材和讲稿，但共享写操作禁用并提供重连。
- 所有课堂写操作携带幂等键和 `expectedRevision`；冲突返回最新快照，不静默覆盖。

### 6.3 教师与投屏操作

教师授课台根据状态只呈现一个主操作：开始本课、下一页、发起活动、查看提交、启动测试、收卷、匿名讲评、继续授课或返回工作台。上一页、暂停、结束、投屏预览作为次级操作；语音播报不能承担翻页职责。

投屏必须提供全屏、上一页、下一页和返回教师端。投屏翻页修改同一 `TeachingCursor`，不能只更新本地画面。讲评只显示错误维度、人数、比例、匿名典型错误和正确推理路径；DTO 中禁止出现学生姓名、ID、个人答案、个人证据或个人成绩。

## 7. 普通课堂活动与正式测试窗口

### 7.1 普通活动

活动窗口：

```text
closed → open → reviewing → closed
```

学生活动状态：

```text
not_started → drafting → submitted → retrying → passed
```

教师端分别显示已开始、已提交、已通过人数。提交不等于通过；只有服务端评价器判断通过，且该活动属于节点必要活动时，才写入微练习通过事实。教师可以结束普通活动，但必须显示未完成学生数；未完成不阻止结束课堂。

### 7.2 正式测试

每个任务的 N02 提供独立正式测试，至少覆盖证据分类、链路重建、错误成果修订和职业结论表达。正式测试状态为：

```text
ready → in-progress ⇄ paused → passed | failed | expired
```

规则：

- 服务端签发 `startedAt` 和 `expiresAt`，标准时长十五分钟。
- 客户端逐题节流保存草稿；刷新恢复同一未过期试卷和剩余时间。
- 客户端不能提交分数，答案和量规只由服务端评分。
- 默认情况下关闭页面后计时继续；显式全班暂停测试时保存统一剩余秒数。
- 到时由任意轮询请求触发幂等收卷，不依赖常驻后台任务。
- 未进入测试的学生显示未作答，不生成零分假成绩。
- 第一名学生提交不能关闭另外两名学生试卷。
- 全部已提交、到时收卷或教师显式提前收卷后，才能进入讲评。
- 0 份提交时 UI 和 API 都禁止讲评；提前收卷必须提示未提交人数。
- 未达 80 分生成四维诊断和精确再学清单；完成规定补强活动后可再次测试，不增加补考审批流程。
- 每个 N02 至少提供 A/B 两套等价材料；第三次连续错误展示完整纠正示例后换题。

## 8. N04 专业成果与教师批阅

### 8.1 成果定义

三个 N04 分别形成：

- P01《室内设备与链路证据表》；
- P02《室外站点与覆盖采集表》；
- P03《投诉信息调查单》。

成果自动引用 N01 至 N03 的结构化活动结果，并记录 `sourceNodeId + sourceAttemptId`。学生可以修改预填值，但不能抹掉来源。内置照片和工单统一标记 `materialOrigin: 'simulated-case'`，页面称为“模拟案例材料”；学生真实作答、填写、提交和教师批阅使用 `actionOrigin: 'user'`。

### 8.2 状态与门禁

界面状态固定为：

```text
编辑中 → 已提交 → 教师退回 → 修订中 → 再次提交 → 教师确认
```

N04 开放后可以保存草稿。正式提交 API 必须同时验证：

1. N04 必要活动真实通过；
2. 本任务 N02 正式测试最高有效分不低于 80；
3. 必填字段通过任务成果 schema；
4. 需要证据的字段已挂接模拟案例证据，或明确登记证据缺口与补证动作；
5. N01 至 N03 上游引用存在且版本有效；
6. `expectedStateRevision` 与当前版本一致；
7. 退回稿至少产生一项字段或证据变更后才能再次提交。

前端禁用只用于解释，API 必须重复执行全部门禁。

### 8.3 教师认证

认证策略写入节点配置：

```ts
type TeacherVerificationPolicy = {
  requiresTeacherVerification: boolean;
  teacherVerificationMinScore: 80;
  minimumCriterionRatio: 0.5;
  rubricCriteria: Array<{ key: string; maxScore: number }>;
};
```

教师批阅页展示成果字段、字段关联证据、字段级批注、N02 分项诊断、V1/V2 差异和完整量规。只有当前版本、所有量规项已填写、总分不低于 80 且每项不低于本项满分 50% 时才能确认。低于阈值只能退回，并必须填写具体反馈；零分、缺项和旧版本确认由 API 拒绝。

批阅独立于课堂生命周期，从教师工作台的待批阅入口进入。课堂结束后仍可退回、比较修订并确认。

## 9. 状态、访问与成绩投影

### 9.1 五条正交状态轴

底层不建立越来越长的万能枚举，而是保存真实事实并投影五条轴：

| 状态轴 | 值 |
|---|---|
| 访问 | `unpublished / locked / open` |
| 学习 | `not-started / in-progress / practice-passed` |
| 正式测试 | `not-required / ready / in-progress / paused / failed / passed / expired` |
| 专业成果 | `not-required / editing / submitted / returned / revising / resubmitted / verified` |
| 能力认证 | `not-reached / pending-review / achieved` |

现有中文状态链继续作为展示投影：

```text
未解锁 → 可学习 → 学习中 → 微练习通过 → 正式测试达标
→ 专业产出已提交 → 待教师复核 → 退回修订 / 教师认证 → 能力达成
```

每个节点由配置声明所需阶段。P1 中 N01/N03 不需要教师认证；N02 以必要活动和正式测试达标为节点达成条件；N04 必须由当前成果版本获得教师认证后才能能力达成。

### 9.2 节点与任务解锁

- N01 必要活动通过后开放 N02。
- N02 必要活动和正式测试达标后开放 N03。
- N03 必要活动通过后开放 N04。
- N04 必要活动通过且发生首次有效成果提交后，本任务形成 `taskAdvanceReady`。
- P02 依赖 P01 的 `taskAdvanceReady`；P03 依赖 P02。
- `output-submitted-once` 是单调历史事实；教师退回后不得重新锁定已开放任务。
- 教师确认只决定 N04 能力达成、任务正式分和成果包交付质量，不阻塞后续学习。

锁定节点被直接访问时只返回前置要求，不加载教材、练习、测试或提交组件。未发布和不存在节点显示各自专用页面，禁止静默重定向到 P1T1-N01。所有写 API 重复执行身份、节点政策、版本和状态门禁。

### 9.3 成绩口径

```text
节点测试最高分 = 本任务 N02 真实正式测试最高有效分
成果量规分 = N04 当前认证版本量规总分
任务综合分 = 节点测试最高分 × 40% + 成果量规分 × 60%
项目综合分 = P01、P02、P03 三个官方任务综合分的等权平均
```

- 正式测试达标但成果未认证时，分别显示测试分和“成果待复核”，不显示任务综合分。
- 任务综合分在当前成果版本认证时冻结，并引用测试尝试 ID、成果版本、审核 ID 和公式版本。
- 后续重测不得静默改变已冻结任务分；重新认证时才产生新冻结版本。
- 三个任务正式分未齐时不显示项目综合分。
- 预置成绩、成果和认证必须标记 `origin: 'demo'`，不得满足真实学生的完成或认证条件。

## 10. 项目成果包与能力图谱

三个任务成果首次有效提交后，成果包可以“汇总查看”；三个当前版本全部教师认证后，成果包才标记为“可交付”。成果包必须能打开：

- 学生填写的完整成果字段；
- 每个字段关联的模拟案例证据；
- 教师字段级批注和总体反馈；
- 退回前后的版本差异；
- 正式测试四维诊断；
- 对应能力节点和冻结成绩来源；
- 三个场景中同一证据框架的迁移对照。

能力图谱保留当前形态，只修真实挂接：正式测试节点进入真实测试，专业产出节点进入真实成果，任务分来自冻结记录，能力达成来自真实事件。学生、教师、投屏和图谱快照必须带有：

```text
snapshotVersion
classroomRevision
studentLearningVersion
generatedAt
```

客户端版本滞后时显示“正在同步”，不得拼接新旧快照。人数、提交数、通过数、成绩和状态禁止在组件中写死。

## 11. 教材深度、互动与视觉

### 11.1 人类学习路径

P01 使用“完整示范 → 引导完成 → 独立作答”；P02 撤除完整推理和答案，只保留检查提示；P03 不预标证据类别，由学生自主确定调查范围、证据组合、判断与下一步动作。三课的支架递减必须在内容和界面上肉眼可辨。

每个 N02 保留“问题、看图、步骤、纠偏、练习、产出”六段。P01T1-N02 继续满足完整案例背景、前置知识、术语、带标注设备图、判断依据、逐步推理、至少两个完整示例、两个反例、分层练习、错误反馈、改正路径、迁移任务、专业模板和评价标准。

P02、P03 开课各增加一个不计分的短检索题。P1 结束增加一次无提示综合迁移活动，用于证明学生掌握的是证据框架而非原题答案。

### 11.2 PixiJS 边界

PixiJS 只用于 N02 的高价值链路重建互动，复用现有拓扑和证据链组件，并提供可键盘操作的 DOM 降级版本。范围分类、结构化记录、四态判断和成果表修订使用标准 React 表单、拖放或排序控件。Pixi 不保存权威状态、不自行判分、不承担测试计时。

### 11.3 Image2 视觉收口

沿用现有 Image2 V4 深色工程化语言、图标库、颜色、半径和排版，不重做已经成立的学生首页、教师工作台和能力图谱框架。重点修复：

- 学生学习页过多并列导航与嵌套滚动；
- “看图”只有文字卡而没有真实带标注关系图；
- 教师更多操作菜单在 1440×900 下被裁切；
- 教师、投屏、播放媒体同时出现多个不一致页码；
- 投屏控制拥挤和离线时完全不可操作；
- 平台总览第八阶段在 1440 宽度被裁切，改为清晰的 `4 × 2` 网格；
- 正文、讲师播报条和底部导航互相遮挡。

所有发布视口至少验证 1440×900 和 1920×1080，并支持键盘导航、200% 缩放和 `prefers-reduced-motion`。

### 11.4 工程冗余清理边界

清理遵循“盘点引用 → 自动化证明未使用 → 可逆隔离 → 验收后删除”的顺序。优先处理重复截图、临时浏览器产物、过期构建包、无引用运行时副本和工具缓存；不得手工编辑生成构建输出来伪造清理结果。

当前 `.git/` 是 GitHub 交付链路的一部分，不再删除或重建。数据库、`content/5g/5g.docx`、导入脚本、已验证媒体、当前/上一/最终验收证据均属于受保护对象，只有专用的可逆隔离步骤能够移动，且不得与功能开发混在同一次提交中。

## 12. 错误、安全与隐私

- 匿名写操作返回 401，角色越权返回 403，状态或修订冲突返回 409。
- 正式测试 token 绑定学生、节点、题目版本和测试窗口，答案及评价器不进入客户端 bundle 或公开 API。
- 投屏 DTO 采用字段白名单，自动测试验证不包含个人身份与答案。
- 所有写操作使用幂等键，避免重复开课、重复翻页、重复收卷、重复提交或重复认证。
- SQLite 写入使用显式事务、WAL、外键和忙等待；迁移采用 expand-first，不执行破坏性自动降级。
- SSH 密码只通过临时环境变量或部署秘密提供，不写入仓库、规格、计划、日志或构建产物。

## 13. 验收不变量

1. 页面打开、教师推送、预置分数和成果名称均不能产生完成或认证。
2. 课堂或自学通过同一必要活动后，另一端立即识别同一事实且不要求重复作答。
3. 刷新和重新登录后保留准确自学段落、活动通过数、尝试历史和正式测试草稿。
4. 教师翻页只同步跟随学生；自主学生个人位置不变。
5. 教师、投屏和三个跟随学生在两秒内显示相同 `lessonId/pageId/pageIndex/revision`。
6. 一个学生断线不影响其他人；重连按跟随或自主模式正确恢复。
7. 未通过必要活动、测试未达标、成果字段或证据不完整时，直接调用 API 也不能提交。
8. 首次有效成果提交后下一任务开放；教师退回不重新锁定。
9. 低于 80 分、任一量规项低于 50%、缺量规项或旧版本都不能教师确认。
10. 正式测试刷新不重置倒计时；第一名提交不关闭其他试卷；0 份提交不能讲评。
11. 投屏讲评序列化结果不包含学生 ID、姓名、个人答案、个人证据或个人成绩。
12. 测试未完成不显示测试分；成果未认证不显示任务综合分；三项任务分未齐不显示项目综合分。
13. 模拟案例材料始终可见标识；真实用户行为与预置演示行为永不混用。
14. 锁定、未发布和不存在节点不能通过 URL 或 API 绕过。
15. 同一快照版本下，学生、教师、投屏、项目页和图谱的人数、提交数、成绩、状态完全一致。
16. P01 有完整支架，P02 明显减弱支架，P03 独立综合迁移；非专业人员关闭讲解后仍能解释位置、身份和方向分别需要什么证据以及原因。
17. 教师不另做 PPT 能完成四课时授课；学生能在课堂、自学和混合三种路径中完成闭环。

## 14. 实施纵切面与阶段交付

实施不按“先做完全部后台、再做全部页面”横向拆分，而按可验收纵切面推进：

1. **P01 真实事实纵切面**：统一活动评价器、自学恢复、正式测试计时与草稿、N04 提交门禁、教师退回/确认、P02 单调开放、图谱和三端回流。
2. **P01 真实课堂纵切面**：唯一教学游标、真实浏览器心跳、课堂生命周期、活动与测试窗口、投屏控制和匿名讲评。
3. **P02/P03 迁移纵切面**：复用同一政策和组件，补齐两套一课时授课包、任务专属活动、测试、成果和项目成果包。
4. **产品与视觉收口**：首页双上下文、学习页信息层级、Image2 对齐、平台总览布局、冗余运行时清理和可访问性。
5. **发布验收**：受影响单测、类型检查、结构检查、生产构建、完整 QA 门禁、真实浏览器旅程、GitHub 推送、SSH 分阶段部署和线上四端冒烟。

每个纵切面必须产生可登录、可操作、可回归、可部署的用户可见结果。部署前备份共享 SQLite 数据库及 WAL/SHM；失败时回退应用 release，不对数据库执行破坏性 down migration。

## 15. 非目标

- 补考申请、审批、改分审批和正式学籍归档；
- 多课程、多班级、多租户和高并发基础设施；
- 注册、找回密码、账号管理后台和统一身份接入；
- AI 问答、学生聊天、讨论区或圆桌课堂；
- 真实外业文件上传和出版社完整资源包在线下载；
- P2 至 P6 的教材建设；
- 用游戏机制替代教材正文、教师讲解、正式测试或专业成果。
