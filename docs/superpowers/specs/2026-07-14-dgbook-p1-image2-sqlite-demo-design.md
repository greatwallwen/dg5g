# DGBook P1 Image2 SQLite 样张设计规格

> **2026-07-15 更新：** 本文件保留为架构背景；与 `2026-07-15-dgbook-p1-complete-sample-requirements.md` 冲突的范围、三 N02 深度、评分、演示种子、课堂同步和工程清理决策，均以后者为准。

## 1. 目标

将现有 DGBook Web Demo 重构为一套可持续演示、可验证、可阶段部署的 P1 数字教材样张。样张优先打通学生学习、教师授课和课程能力图谱三条主线，并让三端读取同一套学习、课堂、成绩与认证事实。

核心演示链路：

```text
演示账号登录
  -> 学生学习首页 / 教师授课工作台
  -> P1 项目与 P01/P02/P03
  -> 能力节点自学 / 课堂跟随
  -> 微练习 -> 正式测试 -> 结构化专业产出
  -> 教师退回或认证
  -> 任务与项目分数投影
  -> 《5G网络信息采集成果包》
  -> 能力图谱同步显示
```

## 2. 已确认决策

- 产品定位是高保真样张 Demo，不是正式教务系统。
- 单实例 Next.js 14 服务，远端运行时为 Node.js 20.20.2、pnpm 9.15.0。
- 使用 SQLite WAL；不使用 PostgreSQL、Redis、微服务或多实例部署。
- SQLite 驱动采用 `better-sqlite3`，避免依赖 Node 22+ 的 `node:sqlite`。
- 演示班级固定为 1 名教师和 3 名学生。
- 使用系统内置账号口令；不接学校统一身份认证。
- 浏览器只持有 HttpOnly 会话 Cookie，身份、角色和学生编号由服务端会话派生。
- 学生自学按个人进度依次进入 P01、P02、P03；教师课堂可进入任一已发布任务。
- 上一任务正式测试达标且任务产出已提交后解锁下一任务，不等待教师认证。
- 上游产出退回后不重新锁死已开始的下游任务，但标记“上游待修订”并阻止 P1 最终完成。
- P01、P02、P03 都必须形成完整学习闭环；P1T1-N02 是唯一最高深度教材标杆。
- 有进行中课堂时，学生首页主行动进入课堂，个人自学保留为次级入口；不自动跳页。
- 任务与项目成绩是 Demo 能力认证分，不回写教务系统。
- 采用方案 A：在现有 Next.js 工程内按纵向闭环迭代，不做一次性重写。
- 每个可运行阶段完成后都部署到当前服务器并执行学生、教师、投屏/图谱冒烟验证。

## 3. 当前实现结论

当前线上版本与本地代码一致暴露以下问题：

- 学生和教师登录后都进入 `/course` 完整图谱。
- 登录只写浏览器 `localStorage`，密码未被服务端验证。
- 学习、课堂与设备状态分别存放在三个进程内 `Map`，重启丢失且不能形成事务。
- 教师课堂接口相信客户端声明的角色，学生进度接口相信 URL 中的学生编号。
- 教师端与投屏端分别统计提交人数，产生 21/24 与 18/24 冲突。
- 教师授课页显示“课堂助手未启动”，同时仍展示课堂状态变化。
- P03 已有源教材、动画和游戏资产，但 Web Demo 类型、图谱和路由没有正确接入。
- 锁定节点可通过 URL 直接打开，未知节点静默回退 P1T1-N01。
- P1T1-N02 有六段导航，但内容模型不足以承载完整教材正文。
- 自学与课堂跟随使用不同内容源，却共享或覆盖游标。
- 微练习通过被错误折算为节点测试 100 分；教师认证会把产出默认记为 100 分。

## 4. 范围

### 4.1 必须交付

- 真实服务端登录与角色路由。
- 学生学习首页和教师授课工作台。
- P1 三任务、十二节点、跨任务顺序和项目成果包。
- P1T1-N02 深度教材正文与交互。
- 自学和课堂跟随两套渲染模式与独立游标。
- 唯一学习状态机、统一节点政策和服务端状态迁移。
- URL 与 API 双层门禁。
- SQLite 单一事实源和四端统一读模型。
- SQLite 课堂会话、命令、设备心跳与回执。
- SSE 失效通知与低频轮询降级。
- Image2 V4 视觉对齐和桌面优先高保真实现。
- 3 名演示学生的确定性事件种子和受控重置。
- 每阶段远端部署、远端冒烟和最终验收。

### 4.2 非目标

- 多课程、多班级、多租户。
- 注册、找回密码、账号管理后台。
- 学校统一身份、教务成绩回写。
- 补考、改分审批、成绩锁定和正式学籍归档。
- Redis、WebSocket 网关、水平扩容和高可用数据库。
- AI 问答、聊天导师、圆桌讨论和多讲师模式。
- P2-P6 学习内容建设。
- 移动端专项视觉精修。

## 5. 技术架构

### 5.1 分层

```text
课程静态定义
  MDX / 媒体 / 节点政策 / Image2 视觉契约
                     |
                     v
Next.js Server Components + Route Handlers
  鉴权 -> 命令服务 -> SQLite 事务 -> 投影服务
                     |
          +----------+-----------+
          |                      |
          v                      v
统一角色读模型              SSE 版本失效通知
  student/teacher/           客户端收到后重新 GET
  projector/graph
```

课程结构、教材正文、媒体和节点政策继续版本化保存在代码库中。账号、会话、学习事件、测试尝试、专业产出、复核、课堂位置、设备心跳、课堂命令和回执存入 SQLite。

### 5.2 数据库运行约束

- 数据库文件默认位于部署共享目录，不放入带时间戳的 release 目录。
- 启用 `journal_mode=WAL`、`foreign_keys=ON`、`busy_timeout=5000`。
- 所有业务写入使用显式事务。
- 学习事件 `event_id` 唯一，正式测试 `attempt_id` 唯一。
- Schema 使用版本化迁移；seed 使用稳定 ID 和幂等 upsert。
- 发布前备份数据库文件及 WAL/SHM，部署后执行迁移和完整性检查。

### 5.3 身份边界

- `users` 保存教师和学生账号、口令哈希、显示名与状态。
- `classes` 保存唯一演示班级。
- `class_memberships` 保存 1 名教师和 3 名学生的角色关系。
- `auth_sessions` 保存随机会话令牌摘要和过期时间。
- 口令使用 Node `crypto.scrypt` 哈希，令牌使用密码学随机数生成。
- Cookie 使用 `HttpOnly`、`SameSite=Lax`，生产环境使用 `Secure`。
- 所有 Server Component 和 API 先执行 `requireUser()` 或 `requireClassRole()`。
- 客户端的 `role`、`studentId`、自定义角色请求头都不再具备授权意义。

## 6. 领域模型与状态

### 6.1 唯一学习状态

```text
未解锁
 -> 可学习
 -> 学习中
 -> 微练习通过
 -> 正式测试达标
 -> 专业产出已提交
 -> 待教师复核
 -> 退回修订 -> 待教师复核
 -> 教师认证
 -> 能力达成
```

节点政策显式配置：

- `requiresMicroPractice`
- `requiresFormalTest`
- `assessmentRole: none | node-test | task-pixi`
- `formalPassScore`
- `requiresProfessionalOutput`
- `requiresTeacherVerification`
- `publicationStatus`
- `prerequisites[]` 及其满足条件

普通节点可依据配置跳过不需要的阶段。三个任务 N04 要求任务级正式测试、专业产出和教师认证。每个任务 N02 承担节点正式测试；微练习不进入正式成绩。

### 6.2 跨任务解锁

- 同任务节点按前一节点 `achieved` 解锁。
- P1T2-N01 需要 P1T1-N04 正式测试达标且首次专业产出已提交。
- P1T3-N01 需要 P1T2-N04 正式测试达标且首次专业产出已提交。
- 上游产出被退回时，下游已解锁状态保持，但项目投影显示上游待修订。
- 三份任务产出全部教师认证后，P1 才能进入完成状态。

### 6.3 成绩口径

- 节点正式测试最高分：仅统计 `assessmentRole=node-test` 的正式尝试。
- 任务 Pixi 正式测试分：仅统计 `assessmentRole=task-pixi` 的最好成绩。
- 任务暂估分：节点正式测试平均分 20% + 任务 Pixi 正式测试 30% + 专业产出量规分 50%。
- 任务正式分：教师认证后冻结的任务综合分。
- 项目综合分：P01、P02、P03 三个正式任务分等权平均。
- 正式测试 100 分不等于教师认证或能力达成。
- 图谱、学生和教师页面不得使用无上下文的“成绩”。

## 7. 路由与权限门禁

服务端在读取教材正文前将节点分为：

- `open`：已发布且当前学生满足前置条件。
- `locked`：已发布但前置条件未满足，只返回前置节点与完成要求。
- `not-open`：节点存在但未发布。
- `not-found`：节点不存在或 ID 格式非法。

锁定、未开放和不存在页面都不得加载教材正文、练习配置、正式测试或提交组件。所有写 API 重复执行同一门禁，禁止只保护页面。任何未知节点不得重定向到 P1T1-N01。

## 8. 角色入口

### 8.1 学生首页 `/student/home`

首屏必须直接回答：

1. 我正在学什么。
2. 为什么学。
3. 下一步做什么。
4. 做到什么算完成。

首页提供当前项目、当前任务、当前节点、完成标准、唯一主行动、其他任务和课程能力图谱入口。存在进行中课堂时，主行动为“进入正在进行的课堂”，个人自学入口保留为次级行动。

学生首页读模型包含：

- 学生和课程。
- 当前项目、任务与节点。
- 岗位化学习价值。
- 配置化完成标准。
- 权威下一行动及原因。
- 个人恢复位置。
- 进行中课堂提示。
- P01/P02/P03 状态和项目成果包状态。

### 8.2 教师工作台 `/teacher/workbench`

工作台显示当前课程、演示班级、最近授课位置、继续授课、开始新课、待批阅产出、班级薄弱点、课堂状态和图谱入口。

- “继续授课”一次点击进入最近位置。
- “开始新课”在当前页打开任务/节点选择器，第二次点击进入 P1T1-N02。
- 工作台不加载完整讲稿、播放器或授课控制条。
- 所有统计来自 3 名学生的统一服务端读模型。

## 9. P1 项目与教材内容

### 9.1 项目链

```text
P01 室内信息采集
  《室内设备与链路证据表》
        |
        v
P02 室外信息采集
  《室外站点与覆盖采集表》
        |
        v
P03 投诉信息采集
  《投诉信息调查单》
        |
        v
《5G网络信息采集成果包》
```

下游产出引用上游 `artifactId + version`，不复制出第二套独立事实。P03 已有的 MDX、动画、Manim 视频和正式游戏资产必须接入 Web Demo，不重新伪造一套内容。

若需要改变由 `content/5g/5g.docx` 生成的教材内容，必须先修改 `scripts/import-5g-docx.py` 或 `scripts/import_5g/` 模块并重新生成；不得只手改生成后的 MDX、AST 或媒体索引。

### 9.2 P1T1-N02 深度教材

保留“问题、看图、步骤、纠偏、练习、产出”六段，但改为一个连续、可独立学习的章节，包含：

- 案例背景、任务问题和必要前置知识。
- 术语解释和可查询词条。
- 带标注设备图与证据定位图。
- 位置、身份、连接方向三类证据及其判断边界。
- 可复述的逐步推理过程。
- 至少 2 个完整正例与 2 个完整反例。
- 基础、应用、迁移三级练习。
- 按错误原因生成的反馈和改正路径。
- 迁移任务、节点证据记录模板和量规。

N02 的节点证据记录进入 P01 N04 任务产出，但 N02 本身不承担任务级教师认证。

### 9.3 其他节点最低规格

- 1 个连续案例背景和明确问题。
- 3–5 个必要术语。
- 1 幅带标注关系图。
- 3–4 步判断过程。
- 1 个完整正确示例与 1 个反例。
- 1 个可重试微练习及针对性反馈。
- 1 份结构化节点产出。

## 10. 自学与课堂跟随

教材采用一个内容定义、两个渲染器和两个游标：

- 自学渲染器展示完整教材、自由导航、术语、案例、练习、反馈、重试和专业产出。
- 课堂渲染器只展示教师当前讲授单元、课堂位置、教师任务和当前活动。
- `self_study_cursor` 按学生与节点保存个人阅读位置。
- `classroom_cursor` 按课堂会话保存教师位置和修订号。
- 自主学习学生只看到“课堂位置已更新”提示。
- 学生主动选择“一键回到教师当前页”后才切换为跟随。
- 课堂命令不得删除或覆盖个人游标。

## 11. 课堂与实时同步

- 课堂会话、成员模式、设备心跳、命令和回执持久化到 SQLite。
- 教师课堂命令使用 `expectedRevision` 乐观并发控制。
- 控制成功只由真实 `applied` 回执确认。
- Helper 离线时明确显示同步不可用，不用种子学生伪造在线。
- SSE 消息只携带主题、版本和作用域，不携带姓名、成绩、答案或教师脚本。
- 客户端收到更高版本后重新 GET 权威快照。
- SSE 断线时每 15 秒低频轮询；恢复连接后读取最新快照。

## 12. 四端统一读模型

学生、教师、投屏和图谱都从同一 SQLite 读事务获得带 `version` 的快照。角色差异只体现在字段裁剪：

- 学生：本人明细和班级匿名聚合。
- 教师：3 名学生明细、复核和薄弱点。
- 投屏：匿名聚合，不含姓名、学生编号、答案和教师私有脚本。
- 图谱：学生看本人节点状态；教师看班级聚合。

统一字段至少包括参与人数、在线人数、提交人数、达标人数、节点测试最高分、任务暂估分、任务正式分、项目综合分、状态码、课堂修订号和快照版本。

## 13. Image2 视觉设计合同

以下 V4 图片是直接产品设计参考，不是背景图或概念稿：

- `dgbook-image2-login-dark-v4.png`
- `dgbook-image2-capability-graph-dark-v4.png`
- `dgbook-image2-learning-dark-v4.png`
- `dgbook-image2-teacher-dark-v4.png`
- `dgbook-image2-student-follow-dark-v4.png`
- `dgbook-image2-pixi-dark-v4.png`

实现必须保留并深化：

- 深海军蓝工程空间、青色当前焦点、绿色达成、黄色风险、红色故障。
- 真实 SVG 图标、半径不超过 8px、边界清晰的工程面板。
- 当前学习对象或专业交互占据屏幕中心。
- 图谱每条边连接真实源节点与目标节点，标签不与连线重叠。
- 教材阅读区优先保证长文可读性；深色工程舞台承载设备关系和标注图。
- 讲师播报条不遮挡正文。
- 动效只表达信号流、当前焦点、状态迁移、课堂同步和即时反馈。
- 常规微交互 150–300ms，并支持 `prefers-reduced-motion`。
- 不使用紫色主题、随机光球、无意义霓虹、PLC 梯形图或通用英文后台风格。

新学生首页和教师工作台沿用同一设计语言：12 列桌面网格、主任务 Bento 布局、克制玻璃质感和明确的唯一主行动。1440×900 与 1920×1080 为发布验收视口。

## 14. 演示种子

提供两个幂等 seed profile：

- `base`：课程、班级、1 名教师、3 名学生和空学习状态。
- `demo`：在 base 上追加可追溯学习事件、课堂会话、提交、复核和测试尝试。

所有教师统计和图谱状态由 demo 事件投影产生，不在组件中写死数字。受控重置只通过脚本或受保护的 Demo 操作执行。

## 15. 验收

### 15.1 自动化

- 真实 Cookie 登录和角色落点。
- 学生首页四问与课堂优先行动。
- 教师一次继续、两次开始新课。
- P1 三任务、十二节点、结构化产出和项目成果包。
- N02 内容 Schema 基数与无语音独立阅读。
- 自学与课堂双游标隔离。
- 所有状态合法迁移、非法跳级、重试幂等和事务回滚。
- 锁定、未开放、不存在节点的页面与 API 门禁。
- 四端相同 `version`、人数、提交数、分数和状态。
- SSE 断线恢复与 15 秒轮询降级。
- Image2 页面截图对齐、键盘导航、200% 缩放和 reduced-motion。

### 15.2 人工体验

5 名不了解 5G 和能力图谱的受试者中至少 4 名：

- 在 30 秒内进入正确学习入口。
- 能说明位置证据回答“在哪里”。
- 能说明铭牌、型号/序列/槽位证据回答“是谁”。
- 能说明端口标签、连续走线和对端证据回答“连向哪里”。
- 能解释三类证据为什么不能互相替代。

## 16. 阶段部署合同

每个纵向阶段必须执行：

1. 相关单元测试和结构审计。
2. `pnpm web:typecheck`。
3. `pnpm web:build`。
4. 准备 source 或 standalone release。
5. 使用现有 SSH 发布脚本部署到当前服务器。
6. 验证 `dgbook-web` 服务为 active。
7. 对公开 URL 执行学生、教师、投屏/图谱冒烟。
8. 截取与 Image2 对应的 1920×1080 页面证据。
9. 失败时切回上一 release；SQLite 迁移采用 expand-first，不自动破坏性 down migration。

SSH 凭据只能由临时环境变量或 CI Secrets 提供，不得写入仓库、计划、日志或构建产物。
